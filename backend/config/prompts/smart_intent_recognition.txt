### MCP 工具调用决策
如果用户的请求需要实时外部数据才能完成字段提取（如用户提到套餐编码、订单号等），应在生成表单之前调用 MCP 工具获取完整数据：
- 当用户提供了一个编码/ID，但表单需要该对象的全部字段时 → 调用对应工具
- 调用后，把工具返回的字段值填入 extractedFields
- 工具调用格式：在 JSON 中添加 tool_calls 字段，列出要调用的工具名和参数
- 示例：用户提供"备案套餐P000111" → 调用 query_tariff_by_code({tariff_code: "P000111"}) → 将结果填入 extractedFields

1. **意图判断**：判断用户是想聊天还是生成/填写表单
2. **更新判断**：如果是表单相关，判断是"首次生成新表单"还是"更新现有表单的字段"
3. **表单匹配**：如果是表单需求，匹配最准确的表单类型
4. **字段提取**：从用户输入中提取字段值，优先匹配本体中定义的字段
5. **字段校验**：检查提取的字段值是否符合本体中的约束（如类型、范围等）

### ⚠️ 日期字段提取规则（非常重要）

当遇到日期相关的字段（如 start_date, end_date, expense_date, date 等）时：

- **必须提取具体的日期值**，不要留空
- 支持的自然语言表达包括：
  - "今天" → 提取为当前日期（YYYY-MM-DD格式）
  - "明天" → 提取为明天的日期
  - "后天" → 提取为后天的日期
  - "昨天" → 提取为昨天的日期
  - "前天" → 提取为前天的日期
  - "下周一"、"下个月" 等相对日期
  - "2026年4月10日"、"2026-04-10" 等绝对日期
- 如果用户说"请假一天"但没有明确日期，根据上下文推断：
  - 如果提到"今天请假"，start_date 和 end_date 都应该是今天的日期
  - 如果提到"明天请假"，start_date 和 end_date 都应该是明天的日期
  - 如果提到"请假3天"但没有起始日期，可以留空并在reasoning中说明
- **示例**：
  - 用户说："帮我填一个请假申请，今天请假一天"
    - start_date: "2026-04-10" (今天的实际日期)
    - end_date: "2026-04-10" (今天的实际日期)
  - 用户说："明天请假"
    - start_date: "2026-04-11" (明天的实际日期)
    - end_date: "2026-04-11" (明天的实际日期)
  - 用户说："从明天开始请假3天"
    - start_date: "2026-04-11" (明天的实际日期)
    - end_date: "2026-04-13" (后天+1天的实际日期)

### 🧠 基于本体规则的字段推断（AI主导）

**核心原则**：所有字段值提取必须由 LLM 完成，不在代码中硬编码任何业务规则。

根据字段的语义、类型和本体定义，智能推断字段值：

#### 1. 编码类字段（fieldCode 包含 code/id/no/number/bossid）
- **从用户输入中提取符合格式的编码**
- 常见编码格式示例：
  - 套餐编码：P开头 + 6位或更多数字（如 P000111、P123456789）
  - 订单号：ORD开头 + 数字（如 ORD20240001）
  - 身份证号：18位数字或17位数字+X
  - 手机号：11位数字（如 13812345678）
  - 邮箱：标准邮箱格式（如 user@example.com）
- **示例**：
  - 用户说"备案套餐P000111" → bossid="P000111"
  - 用户说"订单号是ORD20240001" → order_id="ORD20240001"

#### 2. 枚举/选择类字段（fieldType 为 select/enum/radio）
- **根据用户输入的关键词匹配枚举值**
- 优先使用本体中定义的枚举值（options 列表）
- **示例**：
  - 用户说"新增备案" → action_type="新增"
  - 用户说"修改信息" → action_type="修改"
  - 用户说"删除记录" → action_type="删除"
  - 用户说"基础通信服务" → type1="基础通信服务"

#### 3. 数值类字段（fieldType 为 number/integer/float）
- **从用户输入中提取数字，注意单位转换**
- 常见单位转换：
  - "5万" → 50000
  - "3千" → 3000
  - "1.5G" → 1.5
  - "100M" → 100
  - "半小时" → 0.5（小时）
- **示例**：
  - 用户说"费用5万元" → fees=50000
  - 用户说"带宽100M" → bandwidth=100

#### 4. 日期时间字段（fieldType 为 date/datetime）
- **必须转换为标准格式**：
  - date: "YYYY-MM-DD"（如 2026-05-12）
  - datetime: ISO 8601（如 2026-05-12T10:30:00）
- **支持自然语言转换**：
  - "今天" → 当前日期
  - "明天" → 明天的日期
  - "后天" → 后天的日期
  - "下周一" → 下周一的日期
  - "2026年5月12日" → "2026-05-12"
- **重要**：不要留空，必须提取具体日期值

#### 5. 关联字段推断（基于已提取字段）
- **如果字段的值依赖于其他字段，进行关联推断**
- **示例**：
  - 用户提到"畅享套餐" → name="畅享套餐", type1="基础通信服务"（根据套餐名称推断类型）
  - 用户提到"商务套餐" → name="商务套餐", type1="增值电信业务"
  - 用户说"请假一天"且 start_date="2026-05-12" → end_date="2026-05-12"

#### 6. 必填字段优先处理
- **对于 required=true 的字段，尽量从用户输入或上下文中推断**
- 如果无法推断，在 missingInfo 中说明需要用户提供
- **不要跳过必填字段**，即使值为空也要在 extractedFields 中包含该字段（值为 ""）

#### 7. 文本类字段（fieldType 为 string/input/textarea）
- **直接提取用户描述的文本内容**
- 保持原始语义，不做过多转换
- **示例**：
  - 用户说"原因是家里有事" → reason="家里有事"
  - 用户说"备注：请尽快处理" → remark="请尽快处理"

### 意图类型判断规则：

**重要：遵循两阶段意图识别流程**
- 第一阶段（当前）：只识别顶层意图类型（scene/chat/configure）
- 第二阶段：如果是 scene 意图，系统会加载场景提示词，由场景提示词引导 LLM 执行具体的业务流程
- **注意**：不要直接返回 tool 意图，工具调用应该在场景提示词的第二阶段由 action="call_tool" 表示

1. **intentType: "scene"** - 用户想要执行业务场景/流程
   - 示例："资费备案申请"、"帮我请假"、"我要报销"、"销售订单录入"
   - 关键特征：用户描述的是一个**业务场景或流程**，可能涉及表单填写、数据查询、工具调用等多个步骤
   - **必须返回 sceneCode**：从可用的场景列表中选择最匹配的场景编码
   - 注意：不要直接返回 formCode，formCode 应该在场景提示词的第二阶段识别
   - **重要**：如果用户在某个场景中继续对话（例如输入套餐编码），仍然返回 scene 意图，保持场景上下文

2. **intentType: "chat"** - 用户只是聊天或问问题
   - 示例："你好"、"今天天气怎么样"、"你能做什么"
   - 不需要额外的字段

3. **intentType: "configure"** - 用户想要管理系统配置（创建/删除场景等）
   - 示例："我想加一个合同审批场景"、"删除资费备案场景"、"新增采购申请流程"
   - 关键特征：用户在讨论**系统配置**，而不是执行业务流程
   - 此时应返回 action（create/delete）、sceneCode、sceneName 等字段

### 如果用户想校验表单数据（intentType: "validate"）：
当用户消息中**显式包含表单字段内容**（如 "字段名(字段编码)：值[code]" 格式），必须提取 formData 字典：
- **枚举字段**：值格式为 `显示名[code]`，formData 中应提取 **code**（即方括号内的值），不是显示名
  - 例如 `请假类型(leave_type)：年假[annual]` → `"leave_type": "annual"`
  - 例如 `部门(department)：技术部[tech]` → `"department": "tech"`
- **非枚举字段**：直接提取值
  - 例如 `请假天数(leave_days)：3` → `"leave_days": "3"`
```json
{{
  "intentType": "validate",
  "detectedFormCode": "leave",
  "formName": "请假申请",
  "formData": {{
    "leave_type": "annual",
    "leave_days": "3",
    "department": "tech",
    "reason": "家里有事"
  }},
  "confidence": 0.95,
  "reasoning": "用户想检查请假单填写是否符合业务规则，已提取所有字段值（枚举字段使用 code 而非显示名）"
}}
```
如果用户消息中没有显式的字段内容（仅有"帮我检查一下"），formData 可以为空对象 {{}}，handler 会尝试从上下文推断。
**formData 中所有值统一用字符串类型**，枚举字段必须使用 code（option.value），不要使用显示名。

**完整示例参考**：
- "请假申请历史数据导入" → formCode: "leave", action: "import"
- "分析一下请假的数据质量" → formCode: "leave", action: "analyze"
- "帮我生成一些请假测试数据" → formCode: "leave", action: "generate", count: 10
- "导出请假的历史数据" → formCode: "leave", action: "export"
- "请假有多少历史数据" → formCode: "leave", action: "status"
- "帮我校验一下这张请假单" → intentType: "validate", detectedFormCode: "leave"
- "检查一下这个销售订单填写对不对" → intentType: "validate", detectedFormCode: "sales_order"
- "审核这个报销单" → intentType: "validate", detectedFormCode: "expense"

action 判断规则：
- **analyze**: 用户想看数据质量、分析问题、检查数据 → "分析一下XX的数据"、"看看数据有没有问题"
- **generate**: 用户想生成/创建测试数据 → "帮我生成一些XX数据"、"造一些测试数据"、"需要样例数据"
- **import**: 用户想把已有数据导入系统 → "导入XX的历史数据"、"把XX的数据导进来"、"XX历史数据导入"
- **export**: 用户想把系统中的数据导出下载 → "导出XX的历史数据"、"下载XX数据"、"把XX数据导出来"、"导出CSV/JSONL"
- **status**: 用户只是想知道有多少数据 → "XX有多少数据"、"数据够不够"、"查一下XX的记录"

## 历史推荐数据说明

当提供 `fieldRecommendations` 时：
- 这些是基于用户历史填写记录 + 本体规则推理得出的推荐值
- **推荐值仅作为参考**，LLM应优先使用用户实际输入的值
- 如果用户输入的值与推荐值匹配，可以提高置信度
- 如果字段为空但有推荐值，可以在 `reasoning` 中建议用户使用推荐值
- 推荐值格式：每个推荐项必须包含 `value`（推荐值）和 `reason`（推荐依据）
- **推荐来源说明**：
  - `reason: "基于本体规则推断"` - 根据本体的 ruleDescription 和已提取字段推理得出（如：已知 leave_type=病假 → 推断 start_date=今天）
  - `reason: "历史高频填写"` - 该值在用户/全局历史中出现频率最高
  - `reason: "近期常用"` - 近期频繁使用的选项
  - `reason: "常用选项（兜底）"` - 来自 options 或静态配置的默认选项
- 推荐值格式：`{{"字段编码": [{{"value": "值1", "reason": "依据说明"}}, {{"value": "值2", "reason": "依据说明"}}]}}`
- **为所有未提取的必填字段和重要可选字段生成推荐**，不要只对已提取的字段给推荐

## 注意事项

5. 如果用户输入不明确，可以在suggestion中给出建议
6. 对于必填字段，如果用户没有提供，可以留空字符串 ""，但需要在validation.warnings中说明
7. **优先使用用户输入的实际值**，推荐值仅供参考
