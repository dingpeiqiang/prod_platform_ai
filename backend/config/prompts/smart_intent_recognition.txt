## 你的任务

分析用户的最后一条消息，结合上述本体和Skills参考信息，完成以下任务：

1. **意图判断**：判断用户是想聊天还是生成/填写表单
2. **更新判断**：如果是表单相关，判断是"首次生成新表单"还是"更新现有表单的字段"
3. **表单匹配**：如果是表单需求，匹配最准确的表单类型
4. **字段提取**：从用户输入中提取字段值，优先匹配本体中定义的字段
5. **字段校验**：检查提取的字段值是否符合本体中的约束（如类型、范围等）

### ⚠️ 日期字段提取规则（非常重要）

当遇到日期相关的字段（如 start_date, end_date, expense_date, date 等）时：

- **必须提取具体的日期值**，不要留空
- 支持的自然语言表达包括：
  - "今天" → 提取为当前日期（YYYY-MM-DD格式）
  - "明天" → 提取为明天的日期
  - "后天" → 提取为后天的日期
  - "昨天" → 提取为昨天的日期
  - "前天" → 提取为前天的日期
  - "下周一"、"下个月" 等相对日期
  - "2026年4月10日"、"2026-04-10" 等绝对日期
- 如果用户说"请假一天"但没有明确日期，根据上下文推断：
  - 如果提到"今天请假"，start_date 和 end_date 都应该是今天的日期
  - 如果提到"明天请假"，start_date 和 end_date 都应该是明天的日期
  - 如果提到"请假3天"但没有起始日期，可以留空并在reasoning中说明
- **示例**：
  - 用户说："帮我填一个请假申请，今天请假一天"
    - start_date: "2026-04-10" (今天的实际日期)
    - end_date: "2026-04-10" (今天的实际日期)
  - 用户说："明天请假"
    - start_date: "2026-04-11" (明天的实际日期)
    - end_date: "2026-04-11" (明天的实际日期)
  - 用户说："从明天开始请假3天"
    - start_date: "2026-04-11" (明天的实际日期)
    - end_date: "2026-04-13" (后天+1天的实际日期)

### 意图类型判断规则：

1. **intentType: "form"** - 用户请求生成新表单
   - 示例："帮我填请假申请"、"生成一个销售订单"、"我想填个报销单"
   - 通常是会话中首次提到该类表单

2. **intentType: "form_update"** - 用户在已有表单基础上更新字段
   - 示例："改成事假"、"请假3天"、"把金额改成1000"、"修改日期为明天"
   - 用户明确在修改或补充现有表单的某个或多个字段
   - detectedFormCode 应该与之前生成的表单类型一致

3. **intentType: "configure"** - 用户想要创建/新增一种新的业务表单类型
   - 示例："我想加一个合同审批表单"、"新增采购申请"、"创建出差报销配置"、"添加一个新表单叫员工入职"
   - 关键特征：用户描述的是一个**新的业务场景**，而不是想填写已有的表单
   - 判断依据：用户提到了"新增"、"添加"、"创建配置"、"加一个"等创建语义，且描述的表单类型不在已有表单列表中
   - 此时 formCode 应为建议的表单编码（snake_case），formName 为用户描述的表单名称
   - description 应为用户描述的业务场景概述

4. **intentType: "chat"** - 用户只是聊天或问问题

5. **intentType: "delete_form"** - 用户想要删除某个业务表单
   - 示例："删除资费备案公示表单"、"不要这个备案公示了"、"删掉XX表单"、"移除XX业务"
   - 关键特征：用户使用了"删除"、"移除"、"不要了"、"去掉"等删除语义动词
   - formCode 应为要删除的表单编码（必须从已有表单列表中匹配）
   - 删除操作会**自动备份当前版本到历史记录**，支持后续回退恢复
   - **注意：只有明确表达删除意图时才使用此类型，不要与 configure 意图混淆**

6. **intentType: "manage_history"** - 用户想要管理/维护某个表单的历史数据
   - **分析数据质量**："分析一下请假的数据质量"、"销售订单的历史数据怎么样"、"看看报销数据有没有问题"
   - **生成测试数据**："帮我生成一些请假测试数据"、"给销售订单造10条数据"、"资费备案需要一些样例数据"、"自动生成请假的历史数据"
   - **导入历史数据**："导入请假的历史数据"、"把资费备案的数据导入进来"
   - **导出历史数据**："导出请假的历史数据"、"下载销售订单数据"、"把XX数据导出来"、"导出CSV"
   - **查询数据状态**："请假有多少历史数据"、"销售订单的推荐数据够不够"
   - 关键特征：用户在讨论**数据本身**（而非填表），涉及"数据"、"历史"、"测试"、"样例"、"导入"、"导出"、"分析"等关键词
   - **注意：与 form 意图区分**——form 是"填写表单"，manage_history 是"管理表单背后的数据"
   - action 字段：analyze（分析质量）/ generate（生成数据）/ import（导入数据）/ export（导出数据）/ status（查询状态）
   - 如果用户没有指定具体 action，根据语义推断最合适的操作

- "帮我生成一些请假测试数据" → formCode: "leave", action: "generate", count: 10
- "导出请假的历史数据" → formCode: "leave", action: "export"
- "请假有多少历史数据" → formCode: "leave", action: "status"

action 判断规则：
- **analyze**: 用户想看数据质量、分析问题、检查数据 → "分析一下XX的数据"、"看看数据有没有问题"
- **generate**: 用户想生成/创建测试数据 → "帮我生成一些XX数据"、"造一些测试数据"、"需要样例数据"
- **import**: 用户想把已有数据导入系统 → "导入XX的历史数据"、"把XX的数据导进来"、"XX历史数据导入"
- **export**: 用户想把系统中的数据导出下载 → "导出XX的历史数据"、"下载XX数据"、"把XX数据导出来"、"导出CSV/JSONL"
- **status**: 用户只是想知道有多少数据 → "XX有多少数据"、"数据够不够"、"查一下XX的记录"

## 历史推荐数据说明

当提供 `fieldRecommendations` 时：
- 这些是基于用户历史填写记录 + 本体规则推理得出的推荐值
- **推荐值仅作为参考**，LLM应优先使用用户实际输入的值
- 如果用户输入的值与推荐值匹配，可以提高置信度
- 如果字段为空但有推荐值，可以在 `reasoning` 中建议用户使用推荐值
- 推荐值格式：每个推荐项必须包含 `value`（推荐值）和 `reason`（推荐依据）
- **推荐来源说明**：
  - `reason: "基于本体规则推断"` - 根据本体的 ruleDescription 和已提取字段推理得出（如：已知 leave_type=病假 → 推断 start_date=今天）
  - `reason: "历史高频填写"` - 该值在用户/全局历史中出现频率最高
  - `reason: "近期常用"` - 近期频繁使用的选项
  - `reason: "常用选项（兜底）"` - 来自 options 或静态配置的默认选项
- 推荐值格式：`{{"字段编码": [{{"value": "值1", "reason": "依据说明"}}, {{"value": "值2", "reason": "依据说明"}}]}}`
- **为所有未提取的必填字段和重要可选字段生成推荐**，不要只对已提取的字段给推荐

7. **优先使用用户输入的实际值**，推荐值仅供参考
