## Bug 修复 & 资费备案表单

### LLM Service 空响应崩溃修复

**问题**：`_call_custom()` 中 `content = result.get(...).get('content', '')` 当 API 返回 `"content": null` 时，`get` 拿到 `None` 而非默认值 `''`，后续 `.strip()` 崩溃。
**修复**：改为 `get('content') or ''`，使用 `or` 确保即使值为 `None` 也会 fallback 到空字符串。

### 资费备案表单（tariff_filing）

- 新增 `config/ontologies/tariff_filing.json`（14 个字段）
- 注册场景映射关键词：资费备案、资费、备案、资费申请、资费标准
- 前端 DynamicForm.vue 新增 `radio` 组件支持（备案状态/公示状态/审批状态 使用单选按钮）

### LLM maxTokens 截断修复

**问题**：配置生成 JSON 输出较长（5000-8000 tokens），但默认 maxTokens=2048 导致输出被截断，JSON 解析失败。
**修复**：
- `_call_llm` / `_call_llm_sync` / `_call_custom` 新增 `max_tokens` 参数，支持调用方覆盖默认值
- `admin_service.chat()` 调用时传入 `max_tokens=8192`
- `_call_custom` 新增 `finish_reason=length` 截断检测和 warning 日志

### Nginx 504 超时修复

**问题**：配置生成使用非流式请求（stream=False），MiniMax-M2.7 生成 8192 tokens 需要超过 60 秒，nginx proxy_read_timeout 默认 60s 导致 504。
**修复**：
- `llm_service.py` 新增 `_call_custom_stream_sync()` 方法：使用流式传输（stream=True）同步收集全部内容
- `admin_service.chat()` 对 custom provider 优先使用流式同步调用，绕过 nginx 超时限制

### SSE data 格式兼容性修复

**问题**：MiniMax API 返回 SSE 格式为 `data:{...}`（冒号后无空格），而代码用 `startswith('data: ')` 匹配（要求有空格），导致所有流式数据被跳过。
**影响**：
- `_call_custom_stream_sync`（配置生成）：chars=0
- `_call_custom_stream`（前台流式聊天）：无内容输出
- Anthropic 流式解析：同样受影响
**修复**：全项目统一改为 `startswith('data:')` + `line_text[5:].strip()`，兼容两种格式

### 模型推理（reasoning）内嵌显示到步骤中

**改动**：原来模型推理（💭）是独立的折叠区，现在改为内嵌到对应的处理步骤下方。
- `handleEvent` 中 `reasoning` 事件追加到最近的 `thinking` 步骤的 `step.reasoning` 字段
- 模板中每个步骤下显示「▸ 模型思考 (N 字)」可展开查看
- 移除独立的 `modelReasoningText` / `showModelReasoning` / `toggleModelReasoning`
- 清理 `.model-reasoning-*` CSS（约 50 行）

### configure 意图全链路 reasoning 展示

**问题**：configure 意图调用了 3 次大模型（意图识别 + 配置生成 + 关键词生成），但前端只显示了意图识别那 1 条"模型思考日志"，后 2 次没有。
**原因**：`AdminService.chat()` 和 `generate_scene_keywords()` 使用 `_call_llm()`（不返回 reasoning），而 `_call_llm_sync_with_reasoning` 才能提取模型思考过程。
**修复**：

- `AdminService.generate_scene_keywords()` 同上
- `chat.py` configure 分支 3 处 `_reasoning()` 发送：配置生成 / 关键词生成 / AI 引导对话

### 新业务配置生成缺失 rules 修复

**问题**：AI 生成新业务表单配置时，字段没有 `rules` 校验规则。原因是 `_CONFIG_SYSTEM_PROMPT` 中的字段模板只定义了 5 个属性（fieldCode/fieldName/fieldType/required/options），完全没有 `rules`。
**修复**（二次迭代）：
1. 第一轮：添加 14 种固定 rule_type 枚举 → 用户反馈规则应是 AI 动态生成的，不应限制枚举

   - `admin_service.py` `_CONFIG_SYSTEM_PROMPT`：
     - rules 设计改为「rule_type 由 AI 根据业务语义自由命名」，15 种内置类型仅为推荐
     - 新增 `custom` 类型说明（simple 条件表达式 + lambda）
     - 新增完整的 fieldType 枚举表（11 种：input/textarea/number/select/radio/checkbox/date/datetime/email/phone/file）
     - 新增 `enumConfig` 配置说明（静态枚举 + API 动态枚举）
     - 新增 `submitConfig` 提交配置说明
     - 新增 `defaultValue` 默认值属性
   - `DynamicForm.vue`：
     - `validateForm()` 支持 `custom` 类型（Function 构造沙箱执行 simple/lambda）
     - 未知 rule_type 只记录日志不阻塞表单提交（后端兜底校验）
     - 新增 `datetime`、`email`、`phone`、`file`、`checkbox` 组件渲染
     - 新增 `getFieldOptions()` 方法：优先 enumConfig → 其次 options → API fallback
     - 新增 `handleFileChange()` 文件上传处理

### configure 意图 JSON 解析失败修复

**问题**：配置生成（AdminService.chat）返回的 JSON 被截断，报 `Expecting ',' delimiter: line 118 column 8`。
**根因**：
1. `_call_custom_with_reasoning` 硬编码 `max_tokens=2048`，配置生成 JSON 需要 5000-8000 tokens
2. `_extract_json` 只容错中文引号和尾部逗号，不处理截断 JSON
**修复**：
- `llm_service.py`：`_call_llm_sync_with_reasoning` / `_call_custom_with_reasoning` / `_call_ollama_with_reasoning` 三个方法增加 `max_tokens` 参数
- `admin_service.py`：`AdminService.chat()` 调用时传入 `max_tokens=8192`
- `llm_service.py`：`_extract_json` 新增策略4「截断 JSON 括号补全」，统计未闭合括号/字符串并自动补全
