## 流式输出、日志、表单初始化三项修复（第二轮）

### 问题一：流式输出内容少
- **根因**：`smart_chat_response.txt` 中"回复不超过200字"严格限制了 LLM 输出
- **根因2**：`_call_ollama_stream` 中 `iter_lines()` 同步阻塞在事件循环中，会卡住
- **修复**：去掉200字限制改为"适度简洁"；重构 `_call_ollama_stream` 用线程池 + `asyncio.Queue` 彻底解耦流式读取

### 问题二：日志不清晰
- **根因**：`config_loader.py` 用 `print`，prompt 截断只有 80 字，LLM 原始返回没打印
- **修复**：config_loader 改用 `logging`；同步调用日志截断改为 200 字；`chat.py` stream 接口增加 intent_prompt 前300字、LLM 原始返回前500字、解析后 `intent_type/formCode/fields/confidence` 的日志

### 问题三：表单字段未初始化
- **根因**：前端 `generateFormFromIntent` 调用 `/form/generate` 没传 `extractedFields`，LLM 辛苦提取的字段全部丢失
- **修复**：
  - 前端：`generateFormFromIntent` body 增加 `extractedFields`
  - 后端 schema：`FormGenerateRequest` 增加 `extractedFields: Optional[Dict]`
  - 后端 service：`FormService.generate_form` 增加 `extracted_fields` 参数，优先级 LLM传入 > FieldExtractionSkill > 历史推荐
  - 增加完整日志：生成前打印预提取字段、生成后打印最终填充字段数

## LLM 返回为空问题排查与增强日志

**现象**：用户反馈思考过程显示「LLM 返回为空，降级到 Skills」
**根因分析**：`_call_ollama` 返回空字符串 `''` 时，`if intent_result:` 判定为 False
**可能的实际原因**：
1. Ollama 14B 模型加载超时（首次调用需加载 8-10GB 到内存）
2. 内存不足导致 Ollama 返回空 content
3. 模型被其他进程占用

**改进**：
1. `llm_service.py`：Ollama 返回空 content 时统一返回 `None`（而非空字符串），并记录详细日志（含原始响应）
2. `chat.py`：LLM 调用添加计时，返回为空时显示「耗时 X.Xs」，方便排查超时问题

## LLM 返回为空根因定位

**日志**: `[Ollama] 完整响应=[]`
**排查结果**: 
- Ollama 运行正常，模型 qwen2.5:14b 存在
- 直接测试 Ollama API 返回正常（elapsed=27.7s, content_len=441）
- prompt 长度合理（3351 字符），格式正确

**最终结论**: 最可能的原因是 **uvicorn 热重载**。在 LLM 调用期间（20-30秒）如果文件被修改，uvicorn 会杀掉 worker 进程，导致 requests 请求被中断并返回空。
- 日志也印证了：`WatchFiles detected changes in 'app\api\chat.py'. Reloading...` + `KeyboardInterrupt`

**改进措施**:
1. `llm_service.py`：增加 `done` 状态检查和详细诊断日志（区分「不完整响应」vs「空内容」vs「连接失败」）
2. 建议：修改代码后等后端热重载完成再测试，避免请求被中断
