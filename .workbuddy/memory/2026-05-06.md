## 实现 LLM 智能校验功能（基于 Intent 机制）

用户需求：基于本体配置（ruleDescription）+ 大模型实现规则校验能力

### 设计决策
- **复用 Intent 机制**：通过 `IntentRegistry.dispatch("validate", ctx)` 在提交时触发
- **校验时机**：规则引擎校验通过后 → LLM 智能校验 → 提交
- **校验范围**：规则校验 + 一致性校验 + 业务语义校验

### 新增文件
- `backend/app/intent/handlers/validation_handler.py`：ValidationHandler 实现
  - intent_type = "validate"
  - 基于 ruleDescription 构建 LLM Prompt
  - 调用 llm_service._call_llm() 进行校验
  - 返回 validation_pass/validation_fail 事件

### 修改文件
1. `backend/app/intent/handlers/__init__.py`：注册 ValidationHandler
2. `backend/app/api/form.py`：
   - 导入 IntentRegistry, IntentContext, asyncio
   - submit 端点（第 103 行后）：规则引擎校验成功后调用 LLM 校验
   - 通过 registry.dispatch("validate", ctx) 触发异步校验
   - 解析 SSE 事件获取校验结果，失败则阻止提交

### LLM Prompt 设计
- 传递字段定义（fieldCode, fieldName, fieldType, required, ruleDescription, options）
- 要求校验规则一致性（日期大小、必填、数值范围等）和业务语义合理性
- 输出 JSON 格式：{"valid": bool, "errors": [], "warnings": []}
- 大字段内容截断（超过 500 字）

### 容错策略
- LLM 调用失败/异常：记录日志，跳过校验（不阻塞提交）
- JSON 解析失败：兜底返回 valid=True

## 新增：ValidationSkill 规范化校验架构

**问题**：校验代码散落在 validation_service.py / api/validation.py / validation_handler.py / form_tools.py，职责不清

**解决**：按项目 Skills 规范新增 `ValidationSkill`，统一校验入口

**新增文件**：`backend/app/skills/validation_skill.py`
```python
class ValidationSkill:
    @classmethod
    def validate_field(cls, field_value, rules) -> Dict  # 单字段校验
    @classmethod
    def validate_form(cls, form_data, fields) -> Dict   # 表单校验（规则引擎）
    @classmethod
    def validate_with_ontology(cls, form_code, form_data) -> Dict  # LLM 智能校验
    @classmethod
    def validate_batch(cls, items, rules) -> List        # 批量校验
```

**架构清晰化**：
- `ValidationSkill` - 校验逻辑（规则引擎 + LLM）
- `ValidationHandler` - Intent SSE 事件处理（委托 ValidationSkill）
- `api/validation.py` - REST API（使用 ValidationSkill）
- `form_tools.py` MCP 工具 - 表单提交前校验（规则引擎 + LLM 双检）

**Bug修复 - validate意图 form_code 为 unknown**
- 问题：LLM返回 `detectedFormCode`，但 validation_handler.py 用 `form_code` 读取，导致 form_code=unknown
- 修复：validation_handler.py:29-41 改为兼容多种字段名（form_code/formCode/detectedFormCode）
- 同步修复：chat.py:824 日志字段名兼容性

**前端SSE事件处理补全**
- 问题：ChatAssistant.vue 的 SSE switch/case 没有 validation_fail/validation_pass 分支，导致校验结果事件被静默忽略
- 修复：添加 case "validation_fail"（显示错误）+ case "validation_pass"（显示警告）
- 后端修复：validation_skill 找不到本体时返回 valid=False（原来 valid=True 导致静默通过）

## 重构处理步骤粒度规范

### 规范（三层步骤体系）
- **Phase ≤ 3 个**：每个意图最多 3 个用户可见步骤
- **Phase 必须带 result**：包含该阶段的关键产出数据
- **MCP 工具调用不拆步**：批量执行，结果合并到一个 result
- **错误不单独成 Phase**：错误信息放入 result.success/error
- **reasoning 类型**：专用于 LLM 推理内嵌，不作为独立步骤
