# 2026-05-06 日记

## 优化 form/generate 和 validate/stream 接口日志

两个接口都涉及 LLM 调用，原有日志缺失中间过程，排查困难。修改了 3 个文件：

### 1. app/skills/field_extraction.py
- 新增 `logger = logging.getLogger("field_extraction")`
- `extract()` 方法添加 LLM 调用的输入/输出日志（info + debug 两级）
- 记录：user_input 长度、schema 字段数、LLM 返回成功与否、提取字段数量

### 2. app/services/llm_service.py
- `extract_fields()` 方法添加完整 LLM 调用日志
- 记录：prompt 长度、schema 字段数、原始返回长度、解析结果

### 3. app/services/form_service.py
- `generate_form()` 在调用 `FieldExtractionSkill.extract` 后记录提取方法和字段数
- info 级别记录方法(llm/ontology_based)和提取字段数
- debug 级别记录 LLM 提取结果的完整内容

所有日志输出到统一的 logging 配置（root logger DEBUG，子 logger 继承），与 uvicorn 请求日志输出到同一文件。

## 优化 AI 校验 Loading 体验

优化 DynamicForm.vue 的 AI 校验遮罩层：

### UI 改进
- **AI 脑图动画**：header 中用脉冲圆环 + 🤖 图标替代纯文字，3 层 ring 动画营造"AI 思考中"氛围
- **耗时实时显示**：显示"预计需要 5~15 秒 · 已等待 N 秒"，用户知道要等多久
- **打字机光标**：当前步骤文本前加闪烁光标 `▌`，配合步骤进度
- **动态步骤点**：当前步骤用 3 个弹跳圆点动画替代原来的 `▶`
- **进度条优化**：进度从 `(currentStep / total)` 改为 `((currentStep+1) / total)`，避免初始为 0%

### 数据层改进
- 新增 `elapsedSeconds` 计时器，每秒+1，实时反映等待时长
- 新增 `timerHandle` 管理 `setInterval`，done/error 路径统一清理

### 模型思考日志实时显示
- **后端** (`chat.py` validate/stream)：`_call_llm_sync_with_reasoning` 返回的 `reasoning` 之前被丢弃，现在通过 SSE `reasoning` 类型事件推送给前端
- **前端** (`DynamicForm.vue`)：
  - `aiValidationOverlay.reasoning[]` 数组存储模型思考日志（最多 10 条，超出自动移除最旧）
  - Loading 遮罩下方新增 `llm-reasoning-log` 面板，紫左边框 + 淡入动画实时追加
  - SSE 事件循环中新增 `data.type === 'reasoning'` 处理分支

### validate/stream reasoning 实时推送（修复）
- **问题**：之前的实现用 `run_in_executor` + 阻塞式 `_call_llm_sync_with_reasoning`，reasoning 全部完成后才返回
- **修复**：改用流式 HTTP（`requests.post(stream=True)`）+ `asyncio.Queue` + `SharedResult`
  - 后台线程 `stream_task()` 收 SSE 增量，每收一个 reasoning chunk 立即 `put_nowait` 到队列
  - async generator 用 `await asyncio.wait_for(reasoning_queue.get(), timeout=90)` 轮询
  - reasoning chunks 一到就立即 `yield _sse({"type": "reasoning", ...})` 推给前端（不等 LLM 完成）
  - `SharedResult.content` 在流结束后存入 content，最终解析 JSON
  - 前端每收到一个 reasoning chunk 就 `push` 到 `reasoning[]` 数组实时展示
## 统一意图面板渲染架构

将 ChatAssistant.vue 模板中硬编码的 4 个意图面板（ConfigCard/DeleteResultPanel/HistoryPanel/ValidationResultPanel）重构为基于 intent-registry.js 的动态渲染机制。

### 新增文件
- `frontend/src/components/intent-panels/IntentPanel.vue`：统一动态渲染器，根据 intentType 渲染对应面板，透传所有事件到 ChatAssistant.handleIntentEvent

### 改动文件
1. `frontend/src/composables/useIntentRegistry.js`：新增 `listIntentPanels()` 返回所有已注册面板的 intentType 列表
2. `ChatAssistant.vue`：
   - 4 个 registerEventHandler 统一写入 `msg._intentData[intentType]`
   - 新增 `_updateIntentData(msg, intentType, partial)` helper，统一更新意图数据
   - 新增 `handleIntentEvent` 统一路由所有意图面板事件
   - `loadVersions`/`handleRollback`/`handleConfigDeploy` 改用 `_updateIntentData`
   - 模板：4 个硬编码面板 → `<IntentPanel v-for="intentType in intentPanelTypes" ...>`
   - 新增 `intentPanelTypes = listIntentPanels()` 常量

### 架构原则
- 新增意图类型：**零模板改动** - 只需 `registerEventHandler(type, handler, { panel: MyPanel })` 一行注册
- 意图数据统一走 `msg._intentData[intentType]`，不再有 `msg.configData` 等分散字段
- 事件统一通过 `@intent-action` → `handleIntentEvent` 路由分发
