## 一、意图处理器 Phase 模板规范检查

### 规范要求

根据规范，每个 Handler 的 `handle()` 方法必须遵循以下 Phase 结构：

```python
# ═══ Phase 1：识别 (Identify) ══════════════════════════════════════════
yield thinking("📋 Phase 1：标题 | 详情", result={...})

# ═══ Phase 2：执行 (Execute) ══════════════════════════════════════════
# ── Step 1：子步骤1 ────────────────────────────────────────
yield thinking("📝 Step 1/2：状态 | 详情", result={...})

# ═══ Phase 3：输出 (Output) ══════════════════════════════════════════
yield sse({"type": "stats", ...})
yield intent_event(...)
yield done_event(...)
```

### 问题分析

| 文件 | 问题描述 | 严重程度 |
|------|----------|----------|
| `configure_handler.py` | **Phase 执行顺序错误**：在输出 Phase 1 thinking 之前，已经执行了 `AdminService.chat()` 调用 | **高** |
| `configure_handler.py` | 异常分支缺少 Phase 标记，仅使用 `── 输出错误 ──` 注释 | **中** |
| `delete_form_handler.py` | 成功分支缺少 Phase 2 的 Step 标记 | **低** |
| `delete_form_handler.py` | 异常分支缺少 Phase 标记，仅使用 `── 输出错误 ──` 注释 | **低** |

### 具体问题详情

#### 1. ConfigureHandler - Phase 顺序错误

**位置**：`backend/app/intent/handlers/configure_handler.py:30-135`

**问题**：在输出 Phase 1 的 thinking 之前，已经执行了 `AdminService.chat()` 和时间统计：

```python
# 第30-33行：在 Phase 1 之前执行了核心业务逻辑
loop = asyncio.get_event_loop()
ai_result = await loop.run_in_executor(None, lambda: AdminService.chat(ai_messages))
ctx.stream_stats.total_elapsed = time.time() - ctx.start_time

# 第42行之后才输出 Phase 1
yield thinking(f"🛠️ 识别到新业务配置请求...")  # Phase 1
```

**影响**：违反了规范的 Phase 流程顺序，导致日志记录和实际执行不同步。

**建议修复**：将 `AdminService.chat()` 调用移到 Phase 2 内部。

## 八、建议的修复方案

### 1. ConfigureHandler 修复

将业务逻辑移到正确的 Phase 中：

```python
async def handle(self, ctx: IntentContext) -> AsyncGenerator[str, None]:
    # ═══ Phase 1：识别 ══════════════════════════════════════════
    yield thinking(f"📋 Phase 1：识别到新业务配置请求", result={...})
    
    # ═══ Phase 2：执行 ══════════════════════════════════════════
    ai_result = await loop.run_in_executor(None, lambda: AdminService.chat(ai_messages))
    ...
    
    # ═══ Phase 3：输出 ══════════════════════════════════════════
    ...
```

### 2. thinking 消息格式统一

统一修改所有 Handler 的 thinking 消息格式：

```python
# Phase 1 格式
yield thinking("📋 Phase 1：识别到任务「XXX」", result={...})

# Phase 2 Step 格式
yield thinking("📝 Step 1/2：执行子步骤", result={...})

# 成功/失败格式
yield thinking("✅ 执行完成：结果描述")
yield thinking("❌ 执行失败：错误原因")
```

### 8.2 Schema文件结构规范检查

#### 手册定义的Schema结构

```json
{
  "formCode": "string",
  "formName": "string",
  "description": "string",
  "category": "string",
  "fields": [...]
}
```

#### 实际Schema结构

```json
{
  "formCode": "string",
  "formName": "string",
  "description": "string",
  "entities": [
    {
      "entityCode": "string",
      "entityName": "string",
      "fields": [...]
    }
  ]
}
```

#### 差异对比

| 字段 | 手册定义 | 实际实现 | 说明 |
|------|---------|---------|------|
| `category` | ✅ 必需 | ❌ 缺失 | 实际使用 `entities` 替代 |
| `fields` | ✅ 直接定义 | ❌ 嵌套在 `entities` 中 | 实际采用实体-字段层级结构 |
| `entityCode/entityName` | ❌ 无 | ✅ 存在 | 实际增加了实体概念 |
| `fieldCode` vs `name` | `name` | `fieldCode` | 字段标识命名不同 |
| `fieldType` vs `type` | `type` | `fieldType` | 字段类型命名不同 |
