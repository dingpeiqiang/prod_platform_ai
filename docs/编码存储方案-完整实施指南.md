## 🎯 核心原则

**数据库存储编码，用户界面显示中文**

```
┌─────────────────────────┐
│  UI 显示层              │
│  "年假"、"病假"、"事假"  │  ← 用户看到中文
└───────────┬─────────────┘
            ↓ 映射
┌─────────────────────────┐
│  应用逻辑层             │
│  annual, sick, personal │  ← 代码使用编码
└───────────┬─────────────┘
            ↓ 存储
┌─────────────────────────┐
│  数据库层               │
│  annual, sick, personal │  ← 数据库存编码
└─────────────────────────┘
```

#### 2.2 _get_field_label方法 (第172-218行)

```python
@classmethod
def _get_field_label(cls, form_code: str, field_code: str, code: str) -> str:
    """从本体定义中获取字段的中文标签"""
    try:
        ontology_result = OntologyService.get_form_constraint(form_code)
        if not ontology_result.get('success'):
            return code
        
        ontology = ontology_result.get('constraints', {})
        
        # ⚠️ 本体结构是 entities 数组，需要遍历查找字段
        entities = ontology.get('entities', [])
        for entity in entities:
            fields = entity.get('fields', [])
            for field_def in fields:
                if field_def.get('fieldCode') == field_code:
                    options = field_def.get('options', [])
                    for opt in options:
                        # 支持两种格式：字符串数组 和 {value, label} 对象数组
                        if isinstance(opt, str):
                            if opt == code:
                                return opt
                        elif isinstance(opt, dict):
                            if opt.get('value') == code:
                                return opt.get('label', code)
                    
                    return code
        
        return code
        
    except Exception as e:
        logger.warning("[HistoryService] 获取字段标签失败: %s", e)
        return code
```

**效果**:
- ✅ 推荐接口返回 `{value: "annual", label: "年假"}`
- ✅ 支持新旧两种options格式（向后兼容）

```javascript
// 兼容字符串和对象两种推荐格式
const normalizeRecommend = (rec) => {
  if (typeof rec === 'string') {
    return { value: rec, label: rec, source: 'static', reason: '', confidence: 0 }
  }
  return {
    value: rec.value || '',
    label: rec.label || rec.value || '',  // ⚠️ 优先使用label，如果没有则用value
    source: rec.source || 'history',
    reason: rec.reason || '',
    confidence: rec.confidence || 0
  }
}
```

## 📊 数据流转示例

### 场景：用户导入请假历史数据

#### 1. 准备数据文件 (`leave_test_enum.data.jsonl`)

```jsonl
{"applicant_name":"张三","department":"tech","leave_type":"annual","start_date":"2026-03-15","end_date":"2026-03-20","reason":"回老家探亲","days":5}
{"applicant_name":"李四","department":"sales","leave_type":"personal","start_date":"2026-03-22","end_date":"2026-03-24","reason":"家里有事","days":3}
```

#### 2. 导入过程

```
用户上传文件
    ↓
import_history.py 读取数据
    ↓
flat_for_storage = {"department": "tech", "leave_type": "annual", ...}  # 保持编码
    ↓
写入 FormHistory 表
    ├─ field_code: "leave_type"
    └─ field_value: "annual"  ← 存储编码
    ↓
导入报告统计（使用valueTransform转换显示）
    ├─ leave_type: 年假 (1次), 事假 (1次)  ← 显示中文
    └─ department: 技术部 (1次), 销售部 (1次)
```

#### 3. 推荐过程

```
用户打开新表单
    ↓
前端请求 /api/v1/history/recommend?formCode=leave&fieldCode=leave_type
    ↓
HistoryService.get_recommend_values()
    ├─ 查询 FormHistory: SELECT field_value, COUNT(*) FROM form_history WHERE field_code='leave_type' GROUP BY field_value
    ├─ 结果: [{"value": "annual", "count": 1}, {"value": "personal", "count": 1}]
    ├─ 计算分数并排序
    └─ 调用 _get_field_label() 获取中文标签
        ├─ annual → 年假
        └─ personal → 事假
    ↓
返回: [
  {"value": "annual", "label": "年假", "score": 0.8, "source": "database"},
  {"value": "personal", "label": "事假", "score": 0.8, "source": "database"}
]
    ↓
前端渲染推荐标签
    ├─ 显示: [年假] [事假]  ← 用户看到中文
    └─ 点击"年假" → localFormData.leave_type = "annual"  ← 存储编码
    ↓
用户提交表单
    ↓
后端接收: {"leave_type": "annual", ...}  ← 收到编码
```

## 🎨 优势总结

### 为什么采用编码存储？

| 维度 | 存储中文 | 存储编码 |
|------|---------|---------|
| **数据规范性** | ❌ 可能有歧义（"年假" vs "年 假"） | ✅ 严格一致（永远是"annual"） |
| **国际化** | ❌ 困难 | ✅ 容易（只需改映射） |
| **存储空间** | ❌ 较大（UTF-8） | ✅ 较小 |
| **查询性能** | ⚠️ 一般 | ✅ 索引效率高 |
| **维护成本** | ✅ 简单 | ⚠️ 需要维护映射 |
| **推荐准确性** | ⚠️ 可能有重复 | ✅ 精确统计 |
| **用户体验** | ✅ 直观 | ✅ 通过label实现 |

### 核心优势

1. **数据一致性**: 编码是唯一标识，避免拼写错误
2. **国际化友好**: 同一编码可映射到多种语言
3. **统计分析准确**: 不会因为"年假"和"Annual Leave"被当作不同值
4. **易于扩展**: 新增语言只需添加映射，无需修改数据
5. **用户体验好**: 用户看到中文，系统内部使用编码

## 📝 最佳实践

### 1. 为本体配置options

所有枚举类型的字段都应该在本体中配置options：

```json
{
  "fieldCode": "status",
  "fieldName": "状态",
  "fieldType": "select",
  "options": [
    {"value": "draft", "label": "草稿"},
    {"value": "submitted", "label": "已提交"},
    {"value": "approved", "label": "已批准"},
    {"value": "rejected", "label": "已拒绝"}
  ]
}
```

### 2. 导入数据使用编码

用户准备的历史数据文件应该使用编码：

```jsonl
{"status":"approved","leave_type":"annual"}
{"status":"rejected","leave_type":"sick"}
```

### 3. Schema可选配置valueTransform

如果希望导入报告显示中文，可以在schema.json中配置：

```json
{
  "formCode": "leave",
  "valueTransform": {
    "leave_type": {
      "annual": "年假",
      "sick": "病假"
    }
  }
}
```

**注意**: valueTransform只影响报告显示，不影响数据存储。

### 4. 前端始终使用{value, label}格式

推荐接口返回的数据应该是：

```javascript
{
  value: "annual",   // 用于提交
  label: "年假",     // 用于显示
  score: 0.8,
  source: "database"
}
```

## 🔍 常见问题

### Q1: 旧数据已经存储了中文怎么办？

**A**: 有两种方案：

1. **迁移脚本**: 编写脚本将中文转换为编码
2. **渐进式迁移**: 新数据使用编码，旧数据保持不变，推荐时兼容处理

### Q2: 如果本体中没有配置options怎么办？

**A**: `_get_field_label`方法会返回原值（编码），前端直接显示编码。建议为本体的所有枚举字段配置options。

### Q3: 如何支持多语言？

**A**: 在本体中添加多语言映射：

```json
{
  "fieldCode": "leave_type",
  "options": [
    {
      "value": "annual",
      "labels": {
        "zh-CN": "年假",
        "en-US": "Annual Leave",
        "ja-JP": "年次休暇"
      }
    }
  ]
}
```

然后在前端根据用户语言选择对应的label。
