# 会话中表单保存状态分析与优化方案

## 问题分析

### 当前实现状态

经过全面代码分析，发现**会话中的表单状态没有完整保存到数据库**。

#### 1. 当前保存机制

**前端实现（ChatAssistant.vue）：**
```javascript
// 第411-418行：只保存到localStorage
const saveFormState = () => {
  try {
    localStorage.setItem(formStorageKey.value, JSON.stringify({
      formId: currentFormId.value,
      formSchema: currentFormSchema.value
    }))
  } catch {}
}

// 第433-435行：监听表单状态变化
watch([currentFormId, currentFormSchema], () => {
  saveFormState()
}, { deep: true })
```

**加载机制：**
```javascript
// 第420-429行：从localStorage加载
const loadFormState = () => {
  try {
    const raw = localStorage.getItem(formStorageKey.value)
    if (raw) {
      const state = JSON.parse(raw)
      currentFormId.value = state.formId || ''
      currentFormSchema.value = state.formSchema || null
    }
  } catch {}
}
```

#### 2. 存在的问题

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| ❌ 仅保存在localStorage | 🔴 高 | 清除浏览器数据后丢失 |
| ❌ 无法跨设备同步 | 🔴 高 | 换设备后表单状态不保留 |
| ❌ 与会话消息不同步 | 🟡 中 | 消息已入库，表单状态未入库 |
| ❌ 缺少备份机制 | 🟡 中 | localStorage损坏后无法恢复 |
| ⚠️ Schema体积大 | 🟢 低 | 完整Schema可能达到几十KB |

#### 3. 数据流对比

**消息保存流程（已完善）：**
```
用户/AI消息 → saveMessage() → 后端API → chat_messages表 ✅
```

**表单保存流程（不完善）：**
```
表单生成 → saveFormState() → localStorage ❌
                        ↛ 数据库
```

## 解决方案设计

### 方案选择

考虑到表单Schema的复杂性和体积，提供三种方案：

#### 方案A：作为特殊消息保存（✅ 推荐）

**优点：**
- 利用现有的消息持久化机制
- 与消息历史天然关联
- 实现简单，无需新增表结构
- 支持版本历史

**缺点：**
- 增加消息表的数据量
- 需要特殊处理避免显示在聊天区

**实现方式：**
```javascript
// 当表单生成/更新时，创建一条系统消息
await saveMessage(sessionId, {
  role: 'system',
  content: '',  // 不显示内容
  contentType: 'form_state',
  metadata: {
    type: 'form_state_update',
    formId: currentFormId.value,
    formCode: currentFormSchema.value?.formCode,
    formName: currentFormSchema.value?.formName,
    timestamp: Date.now()
  }
})
```

**加载时：**
```javascript
// 从消息列表中查找最新的form_state消息
const formStateMsg = messages.find(m => 
  m.contentType === 'form_state' && 
  m.metadata?.type === 'form_state_update'
)
if (formStateMsg) {
  // 根据formId重新获取表单Schema
  await fetchFormSchema(formStateMsg.metadata.formId)
}
```

#### 方案B：扩展session_metadata字段

**优点：**
- 与会话直接关联
- 查询效率高
- 一个会话只有一个表单状态

**缺点：**
- session_metadata是JSON字段，不适合存储大对象
- 不支持表单版本历史
- 修改频繁可能导致并发问题

**实现方式：**
```python
# 后端chat_service_v2.py
def update_session_form_state(session_id, form_state, db):
    session = db.query(ChatSessionV2).filter(...).first()
    if session.session_metadata is None:
        session.session_metadata = {}
    session.session_metadata['current_form'] = {
        'formId': form_state.formId,
        'formCode': form_state.formCode,
        'updatedAt': datetime.now().isoformat()
    }
    db.commit()
```

#### 方案C：新建form_sessions表（❌ 不推荐）

**优点：**
- 数据结构清晰
- 支持完整的表单生命周期管理
- 可以存储更多表单相关信息

**缺点：**
- 需要数据库迁移
- 增加系统复杂度
- 维护成本高

**表结构设计：**
```sql
CREATE TABLE form_sessions (
    id SERIAL PRIMARY KEY,
    session_id VARCHAR(64) REFERENCES chat_sessions(session_id),
    form_id VARCHAR(50) NOT NULL,
    form_code VARCHAR(100) NOT NULL,
    form_schema JSONB NOT NULL,
    status VARCHAR(20) DEFAULT 'active',  -- active/completed/cancelled
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP,
    completed_at TIMESTAMP
);
```

## 推荐实施方案：方案A（改进版）

### 核心思路

**不在数据库中存储完整的formSchema**，而是：
1. 保存表单的引用信息（formId、formCode等）
2. 需要时通过formId从后端重新获取Schema
3. 利用已有的FormTemplate和FormInstance表

### 实现步骤

#### Step 1: 保存表单引用到消息metadata

```javascript
// ChatAssistant.vue - generateForm函数中
async function generateForm(config) {
  // ... 调用后端生成表单
  
  // 保存表单引用到metadata
  msg.metadata = {
    ...msg.metadata,
    form_state: {
      formId: result.formId,
      formCode: config.formCode,
      action: 'created',  // created/updated/closed
      timestamp: Date.now()
    }
  }
  
  // 保存到数据库
  await saveMessage(currentDbSessionId.value, msg)
}
```

#### Step 2: 加载时恢复表单状态

```javascript
// ChatAssistant.vue - onMounted或watch dbSessionId
onMounted(async () => {
  // 加载消息
  const dbMsgs = await apiLoadMessages(currentDbSessionId.value)
  messages.value = dbMsgs
  
  // 查找最新的表单状态
  const formStateMsg = [...dbMsgs].reverse().find(m => 
    m.metadata?.form_state?.action === 'created'
  )
  
  if (formStateMsg?.metadata?.form_state) {
    const { formId, formCode } = formStateMsg.metadata.form_state
    // 从后端重新获取表单Schema
    const schema = await fetchFormSchemaByCode(formCode)
    if (schema) {
      currentFormId.value = formId
      currentFormSchema.value = schema
    }
  }
  
  loadFormState()  // 降级：如果数据库没有，尝试localStorage
})
```

#### Step 3: 添加后端API获取表单Schema

```python
# backend/app/api/form.py
@router.get("/form/schema/{form_code}")
async def get_form_schema(form_code: str, db: Session = Depends(get_db)):
    """根据formCode获取最新的表单Schema"""
    template = db.query(FormTemplate).filter(
        FormTemplate.form_code == form_code,
        FormTemplate.is_active == True
    ).order_by(FormTemplate.version.desc()).first()
    
    if not template:
        return {"success": False, "message": "表单模板不存在"}
    
    return {
        "success": True,
        "formCode": template.form_code,
        "formName": template.form_name,
        "version": template.version,
        "schema": template.schema
    }
```

#### Step 4: 前端添加获取Schema的API

```javascript
// frontend/src/services/chatApi.js
export async function getFormSchema(formCode) {
  try {
    const resp = await fetch(`/api/v1/form/schema/${encodeURIComponent(formCode)}`)
    if (!resp.ok) return null
    const result = await resp.json()
    return result.success ? result : null
  } catch (e) {
    console.warn('[chatApi] getFormSchema failed:', e)
    return null
  }
}
```

## 总结

### 当前状态

❌ **会话中的表单状态未保存到数据库**，仅保存在localStorage中。

### 影响范围

- 🔴 清除浏览器数据后表单状态丢失
- 🔴 无法在多设备间同步表单状态
- 🟡 与会话消息的持久化不一致

### 推荐方案

✅ **方案A（改进版）**：将表单引用信息保存到消息metadata，需要时从后端重新获取Schema。

**优势：**
- 无需数据库迁移
- 利用现有基础设施
- 支持版本历史
- 实现相对简单

**实施成本：**
- 后端：添加1个API端点（~30行代码）
- 前端：修改2个函数（~50行代码）
- 总耗时：约2-3小时
