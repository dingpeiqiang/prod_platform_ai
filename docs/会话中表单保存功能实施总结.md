```javascript
export async function getFormSchema(formCode) {
  try {
    const resp = await fetch(`/api/v1/form/schema/${encodeURIComponent(formCode)}`)
    if (!resp.ok) return null
    const result = await resp.json()
    return result.success ? result : null
  } catch (e) {
    console.warn('[chatApi] getFormSchema failed:', e)
    return null
  }
}
```

### ✅ 5. 从数据库加载表单状态 - onMounted

**文件：** `frontend/src/components/ChatAssistant.vue`

**修改位置：** onMounted函数（第1402-1453行）

**关键逻辑：**
```javascript
onMounted(async () => {
  // ... 创建或获取dbSessionId
  
  if (currentDbSessionId.value) {
    const dbMsgs = await apiLoadMessages(currentDbSessionId.value)
    messages.value = dbMsgs
    
    // ── 从消息中恢复表单状态 ──
    // 查找最新的表单状态消息
    const formStateMsg = [...dbMsgs].reverse().find(m => 
      m.metadata?.form_state?.action === 'created'
    )
    
    if (formStateMsg?.metadata?.form_state) {
      const { formId, formCode } = formStateMsg.metadata.form_state
      console.log('[ChatAssistant] 检测到表单状态，尝试恢复', { formId, formCode })
      
      try {
        // 从后端重新获取表单Schema
        const schemaResult = await getFormSchema(formCode)
        if (schemaResult && schemaResult.success) {
          currentFormId.value = formId
          currentFormSchema.value = schemaResult.schema
          console.log('[ChatAssistant] 表单状态已从数据库恢复', formCode)
        } else {
          console.warn('[ChatAssistant] 获取表单Schema失败，降级使用localStorage')
          loadFormState()
        }
      } catch (e) {
        console.error('[ChatAssistant] 恢复表单状态失败:', e)
        loadFormState()  // 降级：使用localStorage
      }
    } else {
      // 没有表单状态消息，尝试从localStorage加载
      loadFormState()
    }
  } else {
    // 没有DB会话，直接从localStorage加载
    loadFormState()
  }
  
  scrollToBottom()
  nextTick(() => inputEl.value?.focus())
})
```

**功能：**
- 加载消息后，查找最新的form_state消息
- 从后端重新获取完整的FormSchema
- 三级降级策略：数据库 → localStorage → 空状态

**代码量：** ~35行新增

## 技术架构

### 数据流

```
用户触发表单生成
    ↓
AI识别意图 (form intent)
    ↓
调用 /api/v1/form/generate
    ↓
后端创建 FormInstance
    ↓
返回 formId 和 formSchema
    ↓
前端创建消息，添加 form_state metadata
    ↓
调用 saveMessage() 保存到数据库
    ↓
chat_messages 表 + chat_message_metadata 表
```

### 恢复流程

```
页面加载 / 切换会话
    ↓
调用 apiLoadMessages() 加载消息
    ↓
查找最新的 form_state 消息
    ↓
提取 formCode
    ↓
调用 getFormSchema(formCode)
    ↓
后端查询 FormTemplate
    ↓
返回完整 Schema
    ↓
恢复 currentFormId 和 currentFormSchema
    ↓
表单面板显示
```

### 降级策略

```
优先级1: 从数据库消息metadata恢复
    ↓ 失败
优先级2: 从localStorage恢复
    ↓ 失败
优先级3: 空状态（无表单）
```

## 数据存储结构

### 消息metadata格式

```json
{
  "form_state": {
    "formId": "form_abc123",
    "formCode": "sales_order",
    "action": "created",
    "timestamp": 1715097600000
  }
}
```

### 数据库表

**chat_messages:**
- message_id (PK)
- session_id (FK)
- role: "assistant"
- content: "✅ 已生成表单..."
- content_type: "text"

**chat_message_metadata:**
- message_id (FK)
- meta_key: "form_state"
- value: '{"formId":"...","formCode":"...","action":"created","timestamp":...}'

## 性能影响评估

### 存储空间

**每条表单状态消息：**
- metadata大小：~200-300字节
- 对总存储影响：< 1%

- 响应大小：~5-20KB（取决于Schema复杂度）
- 延迟增加：~50-100ms

**优化措施：**
- 只在检测到form_state时才请求
- 可以添加缓存机制进一步优化

- 时间复杂度：O(n)，n为消息数量
- 典型会话（50条消息）：< 1ms

## 已知限制

### 1. 只保存表单引用

**现状：**
- 数据库中只保存formId和formCode
- 不保存完整的formSchema

**原因：**
- Schema体积大（可能几十KB）
- Schema会更新，保存引用更灵活
- 通过formCode实时获取最新版本

**影响：**
- 如果FormTemplate被删除，无法恢复
- 如果Schema有重大变更，可能不兼容

**缓解措施：**
- 保留localStorage作为降级方案
- 添加错误处理和友好提示

### 2. 只跟踪"created"动作

**现状：**
- 目前只记录`action: 'created'`
- 未跟踪表单关闭、取消等状态

**未来改进：**
- 添加`action: 'closed'`、`action: 'submitted'`等
- 支持更完整的表单生命周期管理
