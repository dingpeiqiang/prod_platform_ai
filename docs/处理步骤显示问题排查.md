# 处理步骤显示问题排查指南

## 问题描述

用户反馈："正在为你生成 tariff_filing_publicity 表单.." 没有显示"处理步骤"。

## 问题分析

### 当前架构

系统有**两个独立的消息**：

1. **流式消息（SSE）**：包含所有 LLM 交互的处理步骤
   - 意图识别
   - Prompt/Response 展示
   - AI 字段推断
   - 推荐引擎
   - 最终提示："✅ 准备生成 xxx 表单..."

2. **Loading 消息**：前端创建的独立消息
   - 显示："正在为你生成 xxx 表单..."
   - 没有 reasoning 数组
   - 不显示"处理步骤"面板

### 为什么看不到处理步骤？

可能的原因：

1. **消息滚动太快**：流式消息完成后立即创建了 loading 消息，用户还没来得及看
2. **reasoning 数组为空**：后端没有正确发送 thinking 事件
3. **前端渲染问题**：thinking 事件没有被正确添加到 reasoning 数组

## 已实施的修复

### 修复 1：后端添加最终提示（form_handler.py）

在发送 `intent` 事件之前，添加最终的 thinking 步骤：

```python
# 【新增】发送最终提示，告诉用户即将生成表单
yield thinking(f"✅ 准备生成 {form_name or form_code} 表单...")

yield intent_event("form", "generate", ctx.intent_data, is_form=True)
```

这样确保流式消息的最后一个步骤是明确的提示，用户可以清楚地看到处理完成。

### 修复 2：前端 loading 消息不继承 reasoning（useFormHandling.js）⭐

**问题**：loading 消息继承了上一个消息的 reasoning，导致处理步骤重复显示。

**解决**：loading 消息使用空的 reasoning 数组，避免重复：

```javascript
// 【修复】不继承 reasoning，避免重复显示。处理步骤在流式消息中已完整展示
const loadingMsg = {
  id: genId(), role: 'assistant',
  content: `正在为你生成 ${formCode || ''} 表单`, done: true, type: 'chat',
  loading: true,
  loadingDots: 0,
  reasoning: [],  // 空数组，不显示处理步骤
  showReasoning: false
};
```

**效果**：
- ✅ 流式消息显示完整的处理步骤
- ✅ loading 消息不重复显示
- ✅ 用户体验更清晰

### 修复 3：增强推荐引擎的推理步骤展示（form_handler.py）⭐⭐

**问题**：推荐引擎的处理步骤缺少详细的推理过程。

**解决**：添加详细的推荐策略统计和推荐理由摘要：

```python
# Step 1: 添加开始提示
yield thinking("📚 正在查询历史推荐数据...")

# Step 2: 统计各策略使用情况
strategy_summary = {}
for field_code, rec_result in recommendations_result.items():
    strategy = rec_result.strategy_used or "unknown"
    strategy_summary[strategy] = strategy_summary.get(strategy, 0) + 1

# Step 3: 发送详细的推荐结果
yield thinking(
    f"📚 为 {len(all_recommendations)} 个字段生成推荐，共 {total_recs} 条",
    result={
        "fieldCount": len(all_recommendations),
        "fieldSummary": field_summary,
        "totalRecommendations": total_recs,
        "maxPerField": max_recs,
        "strategySummary": strategy_summary,  # 各策略使用统计
        "strategyDetails": strategy_details,   # 策略详情文本
        "processingTimeMs": total_time
    }
)

# Step 4: 发送推荐理由摘要
if all_recommendations:
    reason_lines = []
    for field_code, rec_data in list(all_recommendations.items())[:5]:
        items = rec_data.get("items", [])
        strategy = rec_data.get("strategyUsed", "unknown")
        if items:
            top_value = items[0].get("value", "")
            source = items[0].get("source", "")
            reason_lines.append(f"- {field_code}: {top_value} (来源: {source}, 策略: {strategy})")
    
    reasoning_text = f"📊 推荐结果摘要（前5个字段）:\n\n" + "\n".join(reason_lines)
    yield reasoning_event(reasoning_text)
```

**效果**：
- ✅ 显示推荐策略统计（如：frequency: 3个字段, user_personalized: 2个字段）
- ✅ 显示每个字段的推荐值和来源
- ✅ 显示处理耗时
- ✅ 只展示前5个字段，避免信息过载

## 常见问题

### Q1: 看到了"处理步骤"，但是空的？

**原因**：reasoning 数组存在但没有内容

**解决**：检查后端是否正确发送了 `thinking` 类型的事件

### Q2: 完全没有看到"处理步骤"面板？

**原因**：reasoning 数组为空或不存在

**解决**：
1. 检查前端是否正确处理了 `thinking` 事件
2. 检查 `msg.reasoning.push()` 是否被调用

### Q3: 看到了处理步骤，但很快消失了？

**原因**：流式消息完成后，前端立即创建了新的 loading 消息

**解决**：这是预期行为。流式消息会保留在聊天历史中，用户可以向上滚动查看。

### Q4: Prompt/Response 没有显示？

**原因**：后端没有发送 `reasoning` 类型的事件

**解决**：
1. 确认代码中调用了 `yield reasoning(...)`
2. 确认导入语句正确：`from ..utils import reasoning as reasoning_event`

## 进一步优化建议

### 方案 1：合并消息（推荐）

修改前端 `generateForm` 函数，不创建新的 loading 消息，而是在当前流式消息中继续显示：

```javascript
// 不创建新消息，而是更新当前消息
msg.content = `正在为你生成 ${formCode || ''} 表单...`;
msg.loading = true;
```

**优点**：
- 用户在一个消息中看到完整流程
- 不会造成消息混乱

**缺点**：
- 需要较大改动
- 可能影响其他功能

### 方案 2：保持当前设计（已实施）

保留两个独立消息，但确保流式消息的处理步骤清晰可见。

**优点**：
- 架构清晰
- 易于维护

**缺点**：
- 用户可能需要滚动查看

### 方案 3：自动展开处理步骤

在流式消息完成后，自动展开 reasoning 面板：

```javascript
// 在 done 事件中
msg.showReasoning = true;
```

**优点**：
- 用户无需手动点击
- 立即看到处理步骤

**缺点**：
- 可能占用太多空间
